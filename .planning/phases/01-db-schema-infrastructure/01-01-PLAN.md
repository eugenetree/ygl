<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Bot process → DB | Bot process writes `desiredState` (control signal) into `scraper_control`. Currently in-process (no network), but the column is the IPC seam. |
| Scraper process → DB | Scraper process writes `actualState` and `heartbeatAt` into the same row. |

For Phase 1, both writers are still inside the same Node process (the split happens in Phase 2/3). The threat model focuses on schema-level mistakes that would expose the IPC contract to corruption later.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | `scraper_control.actualState` written by an unintended writer | accept | Application-layer convention only — repository class boundary in Plan 02 enforces "scraper writes actual, bot writes desired". Postgres-level RLS is overkill for a single trusted backend. |
| T-01-02 | Tampering | Multiple rows accidentally inserted into `scraper_control` | mitigate | Migration seeds exactly one row via single-object `values({...})` (NOT an array). Plan 02 uses `updateTable` exclusively (no `insertInto` after migration). The single-row invariant is enforced by code discipline — documented in repository file headers in Plan 02. |
| T-01-03 | Tampering | Wrong column overwritten (e.g. `setActualState` clobbering `desiredState`) | mitigate | Each repository method in Plan 02 sets ONLY its owned columns. Verified in Plan 02's acceptance criteria via grep on `.set({...})` calls. |
| T-01-04 | Information Disclosure | State values exposed to logs / Telegram messages | accept | State values are non-sensitive operational strings (`RUNNING`, `STOPPED`, etc.). No PII; no credentials; intentionally surfaced to bot operator. |
| T-01-05 | Denial of Service | Migration takes a lock that blocks app boot | accept | Single `CREATE TABLE` + single-row `INSERT` is sub-millisecond. No existing tables are altered. |
| T-01-06 | Repudiation | No audit trail for who wrote `desiredState` | accept | Single-operator deployment; bot is the only writer. `updatedAt` provides a coarse "when" trail. Per-actor audit deferred to v2. |
| T-01-07 | Elevation of Privilege | Scraper bypasses bot and writes its own `desiredState` | accept | Convention-enforced via separate repository classes (Plan 02). Both processes share DB credentials in this milestone; finer-grained DB roles deferred. |
| T-01-08 | Spoofing | DB connection spoofed | accept | Connection uses existing `dbClient` config (env-var driven, inside Docker network). No new attack surface. |

**Residual risk:** The single-row invariant is convention-enforced (no `UNIQUE` constraint on a synthetic "always one row" column). Acceptable for this milestone; if Plan 02's repository code accidentally inserts a second row, both writes will hit both rows on the next UPDATE — caught at integration test time in Plan 02.
</threat_model>
